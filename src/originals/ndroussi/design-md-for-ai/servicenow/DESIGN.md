# ServiceNow Design System

> Enterprise workflow design with customizable structure, strong branding hooks, and pragmatic platform clarity. ServiceNow’s modern theming model uses global styles, brand color palettes, typography controls, and shape settings to build consistent experiences across portals and applications.

---

## 1. Visual Theme & Atmosphere

### Overall Aesthetic
ServiceNow’s design language is about **platform-wide consistency for workflow software**. It supports many use cases, but the system is anchored by configurable theme primitives: primary, secondary, and neutral palettes, global typography, shapes, and imagery.

### Mood & Feeling
- **Operational**: workflows should feel orderly and dependable
- **Themeable**: the system is designed to absorb brand identity cleanly
- **Modern**: UI Builder and Theme Builder aim for current, flexible interface styling
- **Accessible**: core tokens and autogenerated UI colors are designed to work together
- **Pragmatic**: design choices should help work get done, not attract attention to themselves

### Design Density
**Medium density**. ServiceNow portals and workspaces carry many controls and widgets, but the system relies on predictable global styles to keep them coherent.

### Visual Character
- Brand color palette organized into primary, secondary, and neutral
- Portal and workspace theming driven by global tokens
- Rounded enterprise cards and contained workflow surfaces
- Layouts often organized by widgets, records, lists, and task forms

## 3. Typography Rules

### Font Stack

```css
/* Theme Builder global typography */
--font-sans: Lato, 'Source Sans Pro', 'Helvetica Neue', Arial, sans-serif;
--font-mono: 'SF Mono', 'Roboto Mono', Menlo, monospace;
```

### Type Scale

| Element | Size | Weight | Line Height | Letter Spacing | Usage |
|---------|------|--------|-------------|----------------|-------|
| Page Title | 32px | 700 | 1.15 | -0.015em | Workspace title |
| Section Title | 22px | 600 | 1.2 | -0.01em | Widget / section heading |
| Card Title | 18px | 600 | 1.3 | 0 | Panel title |
| Body | 16px | 400 | 1.55 | 0 | Main workflow content |
| Small Body | 14px | 400 | 1.5 | 0 | Secondary content |
| Label | 14px | 600 | 1.3 | 0.01em | Buttons, field labels |
| Caption | 12px | 500 | 1.4 | 0.01em | Helper and metadata |

### Typography Philosophy
ServiceNow typography should be **clear, configurable, and serviceable at scale**. It needs to support many custom experiences while staying readable under varied theming.

### Practical Rules
- Keep global typography consistent across portals and workspace pages.
- Use titles, subtitles, body, and captions from the theme system rather than custom one-offs.
- Avoid excessive type variety inside workflow tools.

## 5. Layout Principles

### Global Styles Model
Theme Builder centralizes **color, typography, shape, and imagery** so experiences stay consistent.

| Token | Value | Usage |
|-------|-------|-------|
| `--space-2` | `8px` | Tight spacing |
| `--space-4` | `16px` | Core spacing |
| `--space-5` | `24px` | Card spacing |
| `--space-6` | `32px` | Section spacing |
| `--shape-radius` | `10px-16px` | Workflow-friendly rounded surfaces |

### Layout Rules
- Build pages from widgets and logical content groups.
- Keep branding consistent at the portal or workspace theme level.
- Use neutral spacing and clear record/task boundaries.
- Let form fields, lists, and record details follow shared rhythm.

## 6. Depth & Elevation

### Surface Hierarchy

| Level | Treatment | Use |
|-------|-----------|-----|
| Base | soft neutral background | App or portal shell |
| Card | white bounded widget | Default content grouping |
| Overlay | white elevated layer | Modals / contextual tools |
| Brand shell | themed header / navbar | Portal identity |

### Shadow System

```css
.shadow-card {
  box-shadow: 0 6px 14px rgba(27, 36, 48, 0.06);
}

.shadow-overlay {
  box-shadow: 0 18px 32px rgba(27, 36, 48, 0.14);
}
```

### Depth Rules
- Use theme-level shape and depth rules consistently.
- Keep widget boundaries clear.
- Avoid adding custom shadows that conflict with the global style model.

---

## 7. Do's and Don'ts

### Do
- Define brand colors globally through Theme Builder.
- Use consistent typography and shape tokens across pages.
- Keep workflow modules clear, bounded, and repeatable.
- Let autogenerated UI colors do their job before overriding them.

### Don't
- Theme each widget as a separate visual system.
- Overuse custom colors without tying them to the global palette.
- Mix unrelated corner radii or typography scales.
- Sacrifice workflow clarity for visual novelty.

---

## 8. Responsive Behavior

### Responsive Rules
- Let widgets stack cleanly on smaller screens.
- Preserve primary action visibility within task and record flows.
- Keep portal headers compact but recognizable.
- Maintain large enough controls for touch without overinflating dense workflows.

### Mobile Character
ServiceNow mobile layouts should still feel structured and reliable. The theme system should hold together even when the number of visible widgets is reduced.

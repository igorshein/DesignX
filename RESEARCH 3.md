# Инструкции для ИИ‑агента: полный пакет skills + DESIGN.md для вашего агрегатора по дизайну приложений

Я исследовал лучшие практики оформления skills и DESIGN.md на GitHub. Ниже — сначала **краткие выводы исследования** (чтобы вы понимали, что именно я взял из лучших репозиториев и за счёт чего пакет получается сильнее, чем Claude Design / Open Design), а затем — **готовый набор файлов**, которые вы можете целиком скопировать в репозиторий и передать любому агенту (Claude Code, Cursor, Codex, Gemini CLI).

---

## Часть 1. Выводы исследования: что делает skills и DESIGN.md «лучшими» на GitHub

### Эталонные репозитории и что из них взято

| Источник | Что я взял для вашего пакета |
|---|---|
| **anthropics/skills → frontend-design (SKILL.md)** | Формат: YAML‑frontmatter (`name`, `description`, когда применять), запрет «AI slop» (Inter, фиолетовые градиенты, шаблонные сетки), требование выбирать смелое эстетическое направление до кодинга |
| **Owl-Listener/designer-skills** (91 skill, 9 плагинов) | Разделение по этапам: research → systems → UI → critique → delivery. Skill = знание, команда = применение |
| **VoltAgent/awesome-design-md** | DESIGN.md как «единый источник правды»: токены, типографика, паттерны, запреты — в чистом Markdown, читаемом моделью без парсинга |
| **VoltAgent/awesome-agent-skills** | Модульность: один skill = одна компетенция; агент подключает только нужные для задачи |
| **obra/superpowers и подобные** | Progressive disclosure: короткий триггер в frontmatter, детали в теле, тяжёлые справочники — в отдельных файлах, подгружаемых по запросу |
| **github/primer, Material 3, Apple HIG** | Семантические токены (роль, а не hex), контраст 4.5:1, размеры тач‑целей, шкала типографики по ролям |

### Почему этот пакет сильнее, чем Claude Design / Open Design

1. **Специфичность домена.** Claude Design и Open Design — универсальные. Ваш пакет заточен под агрегатор: карточки работ, masonry‑галереи, фильтры, коллекции, поиск, превью — это 90% UI таких продуктов, и они описаны явно.
2. **UX‑контур, а не только UI.** В большинстве публичных skills нет блока исследований и критики. Здесь есть полный цикл: research → flow → UI → self‑critique → accessibility audit.
3. **Жёсткие запреты + чеклисты самопроверки.** Лучшие skills на GitHub работают именно за счёт негативных ограничений («никогда не делай X») и обязательного чеклиста перед выдачей результата. Это встроено в каждый skill.
4. **Оркестрация.** Отдельный AGENTS.md говорит агенту, какой skill когда активировать — без этого skills «лежат мёртвым грузом».

### 📄 Файл 1: `AGENTS.md` — мастер‑инструкция

```markdown
# AGENTS.md — Правила работы ИИ-агента над проектом Design Aggregator

## Кто ты
Ты — ведущий продуктовый дизайнер и frontend-инженер проекта:
агрегатора дизайна для приложений (галереи лучших UI/UX-решений,
скриншоты экранов, паттерны, фильтры по платформам и категориям).
Ты работаешь как команда из UX-исследователя, UI-дизайнера,
арт-директора и frontend-разработчика одновременно.

## Обязательный порядок работы (НЕ ПРОПУСКАТЬ ШАГИ)
1. Прочитай DESIGN.md ПОЛНОСТЬЮ перед любой задачей. Это единственный
   источник правды о визуальном языке. Конфликт инструкций решается
   в пользу DESIGN.md.
2. Определи тип задачи и активируй соответствующий skill:
   - Новая фича / неясные требования → skills/ux-research
   - Генерация экрана, компонента, страницы → skills/ui-generation
     + skills/aggregator-patterns
   - Оценка/улучшение существующего дизайна → skills/visual-critique
   - Любая выдача UI → финальная проверка skills/accessibility
3. Перед выдачей результата пройди чеклист самопроверки из
   активного skill. Если хотя бы один пункт провален — исправь
   ДО выдачи, а не после замечания пользователя.

## Глобальные запреты (применяются всегда)
- НИКОГДА не используй шрифты Inter, Arial, Roboto, Open Sans
  как акцентные. Они допустимы только если явно указаны в DESIGN.md.
- НИКОГДА не используй фиолетово-синие градиенты на белом фоне,
  glassmorphism «по умолчанию» и эмодзи вместо иконок.
- НИКОГДА не придумывай цвета и размеры «на глаз» — только токены
  из DESIGN.md. Нет подходящего токена → предложи добавить его
  в DESIGN.md, не хардкодь значение.
- НИКОГДА не выдавай UI без состояний: loading, empty, error,
  hover, focus, disabled — обязательны для каждого компонента.
- НИКОГДА не жертвуй контрастом текста ради эстетики.

## Формат выдачи результата
- Код: семантический HTML + CSS-переменные из токенов DESIGN.md
  (или Tailwind с конфигом на основе токенов) + React при необходимости.
- К каждому решению — 2–4 предложения обоснования: какую проблему
  пользователя решает и на какой принцип/токен опирается.
- Если задача неоднозначна — задай до 3 уточняющих вопросов,
  затем предложи решение с явными допущениями.
```

### 📄 Файл 2: `DESIGN.md` — система дизайна агрегатора

```markdown
# DESIGN.md — Design Aggregator

> Этот файл — единственный источник правды о визуальном языке продукта.
> Любой ИИ-агент и человек обязаны следовать ему. README отвечает на
> вопрос «что делает продукт», этот файл — «как продукт выглядит
> и ощущается».

## 1. Продукт и аудитория
Агрегатор лучших дизайнов приложений: кураторские галереи экранов,
UI-паттернов и флоу реальных продуктов. Аудитория — продуктовые
дизайнеры, frontend-разработчики, фаундеры. Они приходят за
насмотренностью и референсами, поэтому ГЛАВНЫЙ ГЕРОЙ ИНТЕРФЕЙСА —
ЧУЖОЙ ДИЗАЙН (скриншоты работ). Наш UI — нейтральная галерея,
которая не конкурирует с контентом.

## 2. Характер и эстетика
- Направление: editorial / gallery-minimal. Ощущение музея
  современного искусства, а не маркетингового лендинга.
- Хром интерфейса почти монохромный; цвет приходит из контента
  (скриншотов). Один точный акцентный цвет.
- Много воздуха. Сетка строгая. Никаких декоративных градиентов,
  теней «для красоты», стеклянных панелей.
- Тон текстов: уверенный, лаконичный, без восклицаний и маркетингового
  жаргона. Пишем «Сохранено в коллекцию», а не «Ура! Добавлено! 🎉».

## 3. Цветовые токены
Тёмная тема — основная. Светлая — вторичная.

| Токен | Dark | Light | Роль |
|---|---|---|---|
| color-bg-base       | #0A0A0B | #FAFAF8 | фон страницы |
| color-bg-surface    | #141416 | #FFFFFF | карточки, панели |
| color-bg-elevated   | #1C1C1F | #FFFFFF + shadow | модалки, дропдауны |
| color-text-primary  | #F4F4F2 | #161618 | заголовки, основной текст |
| color-text-secondary| #A1A1AA | #6B6B70 | подписи, мета-данные |
| color-text-muted    | #5C5C66 | #9C9CA3 | плейсхолдеры, disabled |
| color-border        | #26262B | #E6E6E2 | разделители, рамки карточек |
| color-accent        | #E8FF47 | #4D5BFF | ссылки-действия, активные фильтры, фокус |
| color-accent-fg     | #0A0A0B | #FFFFFF | текст на акцентном фоне |
| color-danger        | #FF5C5C | #D92D2D | ошибки, удаление |

Правила:
- Акцент используется ТОЧЕЧНО: активный фильтр, primary-кнопка,
  индикатор сохранения. Не более 1–2 акцентных элементов на экран.
- Все пары текст/фон обязаны давать контраст ≥ 4.5:1 (≥ 3:1 для
  текста ≥ 24px). Проверяй перед выдачей.
- Скриншоты работ всегда лежат на color-bg-surface с рамкой
  color-border 1px — чтобы белые и тёмные скриншоты читались одинаково.

## 4. Типографика
- Заголовки/акциденция: характерный гротеск с высоким контрастом
  форм (Space Grotesk / General Sans / аналог). НЕ Inter.
- Текст/UI: нейтральный гуманистический гротеск с хорошей читаемостью
  на малых кеглях (например, Geist / Suisse-подобный).
- Моно (мета-данные, теги, счётчики): JetBrains Mono или аналог —
  это фирменная деталь продукта.

Шкала:
| Роль | Размер/интерлиньяж | Начертание |
|---|---|---|
| display  | 44/48 | 600, трекинг -0.02em |
| h1       | 32/38 | 600 |
| h2       | 24/30 | 600 |
| h3       | 18/24 | 600 |
| body     | 15/22 | 400 |
| caption  | 13/18 | 400, color-text-secondary |
| micro/mono| 12/16 | 500, mono, uppercase, трекинг +0.06em |

## 5. Сетка, отступы, форма
- Spacing-шкала: 4, 8, 12, 16, 24, 32, 48, 64, 96.
- Контейнер: max-width 1440px, поля 32px (desktop), 16px (mobile).
- Галерея: masonry/grid, 4 колонки ≥1280px, 3 — ≥1024px,
  2 — ≥640px, 1 — мобайл. Gap 16px.
- Радиусы: карточки 12px, кнопки/инпуты 8px, теги 999px (pill),
  скриншоты внутри карточек 8px.
- Тени: только у elevated-слоёв (модалки, дропдауны):
  0 8px 32px rgba(0,0,0,.4). У карточек теней НЕТ — только рамка
  и изменение фона при hover.

## 6. Ключевые компоненты

### Карточка работы (главный компонент продукта)
- Состав: скриншот (соотношение сохраняется, объект cover не обрезает
  важное), название приложения (h3), категория (micro/mono),
  кнопка «сохранить» (появляется при hover на desktop, всегда видна
  на touch).
- Hover: фон → color-bg-elevated, рамка → чуть светлее, скриншот
  scale(1.02) за 200ms ease-out. Никаких подъёмов с тенями.
- Вся карточка — одна ссылка; вложенные действия (сохранить) —
  отдельные кнопки с stopPropagation и собственным focus-стилем.

### Фильтры и теги
- Горизонтальная лента pill-тегов под шапкой; активный таг —
  фон color-accent, текст color-accent-fg.
- Счётчик результатов в mono рядом с фильтрами («247 SCREENS»).
- Сброс фильтров — текстовая ссылка, не кнопка.

### Кнопки
- Primary: фон color-accent, текст color-accent-fg, высота 40px.
- Secondary: прозрачный фон, рамка color-border, текст primary.
- Ghost: только текст, для третьестепенных действий.
- Тач-цель любой кнопки ≥ 44×44px.

### Поиск
- Крупное поле в шапке, открывается по «/». Подсказки: последние
  запросы + популярные категории. Пустой результат — см. раздел 7.

## 7. Обязательные состояния
- Loading: skeleton-карточки (анимированный градиент на
  color-bg-surface), НИКОГДА не спиннер на весь экран.
- Empty (нет результатов): короткий заголовок, 1 строка пояснения,
  кнопка сброса фильтров, 3–4 популярных тега как выход. Без
  иллюстраций-клипартов и эмодзи.
- Error: что случилось + что делать + кнопка повтора. Без кодов
  ошибок в лицо пользователю.
- Focus: видимое кольцо 2px color-accent со смещением 2px — ВЕЗДЕ.

## 8. Motion
- Базовая длительность 150–200ms, ease-out. Появление контента —
  fade + 8px translateY, со stagger 30ms для сеток.
- Один «вау-момент» на продукт: анимация раскрытия карточки в
  полноэкранный просмотр (shared-element ощущение). Всё остальное —
  сдержанно.
- Уважай prefers-reduced-motion: отключай transform-анимации.

## 9. Запрещено
- Градиенты в хроме интерфейса; цветные фоны секций.
- Карусели с автопрокруткой.
- Более одного акцентного цвета.
- Эмодзи в UI-текстах и вместо иконок.
- Тени у карточек, glassmorphism, неоновые свечения.
- Скруглять скриншоты работ сильнее 8px и накладывать на них оверлеи,
  ухудшающие читаемость чужого дизайна.
```

### 📄 Файл 3: `skills/ux-research/SKILL.md`

```markdown
---
name: ux-research
description: Use when requirements are unclear, a new feature is requested,
  or before designing any new screen/flow. Produces user context, scenarios
  and screen structure BEFORE any visual work.
---

# UX Research & Flow Design

## Когда применять
Всегда перед ui-generation, если: фича новая, требования размыты,
или пользователь просит «сделай экран X» без контекста.

## Процесс
1. Сформулируй задачу пользователя продукта (Job To Be Done) в одном
   предложении: «Когда я …, я хочу …, чтобы …».
2. Определи персону из DESIGN.md §1 и контекст (устройство, спешка,
   частота использования).
3. Опиши основной сценарий по шагам (entry point → шаги → успех)
   и 1–2 краевых случая (нет данных, ошибка, первый визит).
4. Построй структуру экрана: список зон в порядке приоритета
   (что пользователь должен увидеть первым, вторым, третьим).
5. Перечисли необходимые состояния (loading/empty/error) и данные.
6. Только после этого передавай работу в ui-generation.

## Формат выдачи
Markdown-блок: JTBD → сценарий (нумерованный список) → структура
экрана (приоритизированный список зон) → состояния → открытые вопросы
(максимум 3).

## Запрещено
- Пропускать этот skill «потому что задача простая».
- Придумывать функциональность, которую пользователь не просил,
  без пометки «предложение сверх задачи».
```

### 📄 Файл 4: `skills/ui-generation/SKILL.md`

```markdown
---
name: ui-generation
description: Use when creating any screen, page, or component (HTML/CSS/React).
  Produces production-grade UI strictly following DESIGN.md tokens and
  aggregator-patterns. Always paired with accessibility check before output.
---

# UI Generation

## Перед началом (обязательно)
1. Перечитай разделы DESIGN.md §3–§9.
2. Получи структуру экрана от ux-research (или построй её за 5 строк,
   если задача тривиальна).
3. Назови выбранное композиционное решение одним предложением
   («masonry-галерея с липкой лентой фильтров и крупным заголовком-счётчиком»).

## Правила генерации
- Только токены из DESIGN.md. CSS — через переменные:
  var(--color-bg-surface) и т.д. Tailwind — только с конфигом на токенах.
- Семантический HTML: nav, main, section, button (не div с onClick),
  alt у всех изображений работ — название приложения + тип экрана.
- Каждый компонент сдаётся со ВСЕМИ состояниями из DESIGN.md §7.
- Типографика строго по шкале §4; не вводи промежуточных размеров.
- Иерархия: на экране ровно одна доминанта. Если выделено всё —
  не выделено ничего.
- Motion по §8; обязательно prefers-reduced-motion.

## Чеклист самопроверки (пройти ДО выдачи)
- [ ] Все цвета/размеры — из токенов, хардкода нет
- [ ] Контраст текста ≥ 4.5:1 на всех парах
- [ ] Loading / empty / error / hover / focus / disabled — есть
- [ ] Тач-цели ≥ 44px, фокус-кольцо видно с клавиатуры
- [ ] Ни одного пункта из DESIGN.md §9 «Запрещено» не нарушено
- [ ] Скриншоты работ — главный визуальный акцент, хром не спорит с ними
- [ ] Выдан код + 2–4 предложения обоснования решений
```

### 📄 Файл 5: `skills/aggregator-patterns/SKILL.md`

```markdown
---
name: aggregator-patterns
description: Domain patterns for design-aggregator products - galleries,
  cards, filters, collections, search, detail view. Use together with
  ui-generation for any core product screen.
---

# Паттерны агрегатора дизайна

## Главная / Галерея
- Above the fold: заголовок-утверждение + лента фильтров + первые
  карточки. Никаких hero-секций с маркетингом — пользователь пришёл
  смотреть работы.
- Бесконечная прокрутка с явным маркером подгрузки (skeleton-ряд);
  каждые ~3 экрана — sticky-возможность вернуться наверх.
- Сохраняй позицию скролла при возврате из детального просмотра —
  это критично для опыта «копания в галерее».

## Детальный просмотр работы
- Открывается оверлеем поверх галереи (модалка-панель), URL меняется
  (диплинк работает). Esc / клик мимо / стрелка назад — закрытие.
- Навигация ←/→ между работами без возврата в сетку.
- Состав: крупный скриншот → мета (приложение, платформа, категория,
  дата) в mono → действия (сохранить в коллекцию, скопировать ссылку,
  открыть источник) → блок «похожие» (4–8 карточек).

## Фильтрация и поиск
- Фильтры применяются мгновенно, без кнопки «Применить».
- Активные фильтры всегда видимы и снимаются в один клик.
- Состояние фильтров — в URL (шарабельность — ключевая ценность
  агрегатора).
- Пустой результат: предложи ослабить последний применённый фильтр.

## Коллекции (сохранённое)
- Сохранение — оптимистичное (мгновенный отклик, откат при ошибке),
  с тихим toast «Сохранено» + ссылкой на коллекцию.
- Первая коллекция создаётся неявно («Saved»), управление папками —
  потом, не в момент первого сохранения.

## Запрещено в этом домене
- Заставлять регистрироваться до просмотра контента.
- Watermark-и и оверлеи поверх скриншотов в сетке.
- Открывать детальный просмотр новой страницей с потерей скролла.
```

### 📄 Файл 6: `skills/visual-critique/SKILL.md`

```markdown
---
name: visual-critique
description: Use when asked to review, improve or audit an existing design,
  and as self-review after ui-generation. Produces structured critique
  with prioritized fixes.

# Visual Critique

## Процесс
Оценивай дизайн строго по шести осям, в этом порядке:

1. **Иерархия.** Что глаз видит первым, вторым, третьим? Совпадает ли
   это с приоритетами задачи пользователя? Есть ли ровно одна
   доминанта на экране?
2. **Соответствие DESIGN.md.** Пройди по токенам §3–§5: найди все
   хардкоды цветов, размеры вне шкалы, нарушения §9 «Запрещено».
   Каждое нарушение — отдельным пунктом со ссылкой на раздел.
3. **Контент-фокус.** В агрегаторе главный герой — чужие скриншоты.
   Не спорит ли хром с контентом? Не перекрыты ли работы оверлеями?
4. **Состояния.** Проверь наличие loading / empty / error / hover /
   focus / disabled. Отсутствующее состояние = критический дефект,
   а не «пожелание».
5. **Консистентность.** Одинаковые сущности выглядят одинаково?
   (радиусы, отступы, стили карточек, поведение фильтров на разных
   экранах).
6. **Эмоция и характер.** Похоже ли это на «музей современного
   искусства» из DESIGN.md §2 — или на шаблонный SaaS-лендинг?
   Назови конкретные элементы, разрушающие характер.

## Формат выдачи
Таблица замечаний:

| # | Серьёзность | Проблема | Где | Как исправить | Ссылка на правило |
|---|---|---|---|---|---|

Серьёзность: **Critical** (блокирует выдачу) / **Major** (портит опыт) /
**Minor** (полировка). После таблицы — сводка: 3 главных исправления,
которые дадут 80% улучшения.

## Правила критики
- Каждое замечание обязано содержать конкретное исправление, а не
  только констатацию («контраст 3.1:1 → замени color-text-muted на
  color-text-secondary»).
- Хвали только то, что реально работает, одним коротким абзацем —
  это нужно, чтобы не «исправить» сильные стороны.
- Если дизайн нарушает DESIGN.md, но нарушение объективно лучше —
  не молчи: предложи обновить DESIGN.md отдельным пунктом.

## Запрещено
- Вкусовщина без опоры на принцип или токен («мне не нравится»).
- Переделывать дизайн целиком, когда просили ревью.
- Выдавать список из 30 равнозначных пунктов без приоритизации.
```

```markdown
# Accessibility Audit

## Когда применять
Всегда, как последний шаг перед выдачей любого UI. Это не опция —
это ворота качества.

## Чеклист (все пункты обязательны)

### Контраст и цвет
- [ ] Текст ≤ 24px: контраст ≥ 4.5:1; текст > 24px или 19px bold: ≥ 3:1
- [ ] Нетекстовые элементы (рамки инпутов, иконки-действия): ≥ 3:1
- [ ] Информация не передаётся ТОЛЬКО цветом (активный фильтр =
      цвет + форма/вес, ошибка = цвет + иконка + текст)

### Клавиатура и фокус
- [ ] Весь флоу проходится с клавиатуры: Tab-порядок логичен,
      ловушек фокуса нет
- [ ] Фокус-кольцо видно на КАЖДОМ интерактивном элементе
      (2px color-accent, offset 2px — по DESIGN.md §7)
- [ ] Модалка детального просмотра: фокус заперт внутри, Esc закрывает,
      фокус возвращается на карточку-источник
- [ ] Шорткат «/» для поиска не перехватывает ввод в полях

### Семантика
- [ ] Кнопки = button, ссылки = a, никаких div с onClick
- [ ] У всех изображений работ — осмысленный alt
      («Onboarding-экран приложения Linear, тёмная тема»)
- [ ] Заголовки идут по порядку (h1 → h2 → h3, без пропусков)
- [ ] Иконки-кнопки имеют aria-label; toast — aria-live="polite"
- [ ] Лента карточек размечена как list/listitem или feed

### Touch и motion
- [ ] Тач-цели ≥ 44×44px, расстояние между соседними ≥ 8px
- [ ] Hover-действия (кнопка «сохранить» на карточке) доступны
      на touch-устройствах постоянно
- [ ] prefers-reduced-motion отключает transform/scale-анимации
- [ ] Бесконечная прокрутка не блокирует доступ к футеру/навигации
      с клавиатуры

## Формат выдачи
Если всё пройдено: одна строка «A11y: passed (контраст, клавиатура,
семантика, touch, motion)». Если нет — список проваленных пунктов
с исправлениями, исправить, прогнать чеклист повторно. UI с проваленным
чеклистом НЕ ВЫДАЁТСЯ.
```

## Часть 4. Как передать пакет ИИ‑агенту

### Вариант A: Claude Code / агенты с поддержкой skills
1. Положите файлы в структуру из Части 2 (для Claude Code: `skills/` → `.claude/skills/`, `AGENTS.md` → `CLAUDE.md`).
2. Агент сам подхватит frontmatter‑описания и будет активировать skills по триггерам в `description`.

### Вариант B: Cursor / Codex / Gemini CLI / любой другой
1. `AGENTS.md` оставьте в корне — большинство агентов читают его автоматически (Cursor также понимает `.cursorrules`).
2. В первом сообщении напишите: *«Прочитай AGENTS.md, DESIGN.md и все SKILL.md в папке skills/. Подтверди, что понял порядок работы, и только потом начинай задачу: [ваша задача]»*.

### Вариант C: чат без доступа к файлам
Вставьте содержимое в одно сообщение в порядке: `AGENTS.md` → `DESIGN.md` → пять skills, и добавьте: *«Это твоя постоянная инструкция на всю сессию»*.

### Первая проверочная задача (рекомендую)
> «Спроектируй и сверстай главную страницу галереи: лента фильтров, masonry-сетка карточек работ, поиск по "/", все состояния. Следуй полному циклу: ux-research → ui-generation + aggregator-patterns → visual-critique → accessibility».

Если агент пропустит research или выдаст UI без empty/error‑состояний — значит, он не прочитал инструкции, и нужно явно указать ему на `AGENTS.md`, шаг 2–3.

## Что вы можете доработать под себя

1. **DESIGN.md §3–§4** — подставьте свои реальные цвета и шрифты, когда определитесь с брендом; структуру и правила менять не нужно.
2. **aggregator-patterns** — по мере роста продукта добавляйте паттерны: профили авторов, подписки, уведомления — каждый отдельным разделом по той же схеме (правила + запреты).
3. **Новые skills** — добавляйте по принципу «один skill = одна компетенция»: например, `seo-content` для текстов карточек или `motion-design` если захотите усложнить анимации.

Пакет готов к использованию — можете копировать файлы в репозиторий прямо сейчас.
